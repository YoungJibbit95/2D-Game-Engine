# Engine Status

This is a compact inventory of the current engine state. Open work remains in the checklist files; this file is allowed to mention completed systems for orientation.

## What Exists

The project has a solid MonoGame/.NET 8 shell with a clean `Game.Core` separation. Core already covers chunked world storage, finite and horizontally infinite world bounds, budgeted chunk streaming, chunk metadata, tile coordinate helpers, deterministic world generation, streamed chunk save/load, packed chunk region files, coordinated save/autosave/load orchestration, project/session bootstrapping, persistent equipment/status effects/character appearance, content registries, sprite asset manifests and audits, runtime sprite animation clips, data-driven item actions with typed feedback results/events, crafting, farming, topdown maps, dialogue, shops, loot, combat, projectiles, entities, spawning, settings, engine-neutral UI animation/layout primitives, underground-aware lighting, time, spatial queries, batched tile mutation results, a reusable performance profiler, and a core runtime simulation tick with world-simulation scheduling.

The engine now has a first explicit game-project boundary. `yjse.game.json` describes a concrete game, `Game.Core.Projects` resolves content/mod/save paths, and `Game.Core.Sessions.GameSessionBootstrapper` creates or resumes playable sessions from project root, save directory, seed, world name, settings, startup, and content data. The MonoGame client can run an external game repository via `YJSE_GAME_ROOT` or a loose content root via `YJSE_GAME_DATA`, while local `Game.Data` remains sample/dev content. Startup profiles now own starter inventory, selected hotbar slot, startup map, and default world profile, so concrete game starts are data-driven.

The client now has YjsE branding, a main menu, world-select and create-world screens, a loading state that can resume existing saves, a keyboard/mouse-driven settings menu, an in-game pause menu, a playable state, camera follow, debug overlays, hotbar UI, a Terraria-like inventory overlay with visible slots, cursor stack, quick move, live equipment slots, stat summary, and item tooltips, a crafting overlay with recipe category tabs, recipe list, ingredient/station checks, craft button, and shift-repeat crafting, visible tile/liquid rendering, and mouse-based selected item use for mining, building, melee, ranged, magic casting, and mana consumables. Escape no longer closes the game from the main menu by accident. Resolution, fullscreen, and VSync settings apply live through the client shell. Rendering is no longer only rectangle-driven: player, enemy, dropped item, projectile, inventory, hotbar, crafting result, and ingredient draw paths now resolve real sprite assets where available and fall back safely. There is a first pass at render layers, shader registry scaffolding, post-processing settings, a texture registry that can resolve sprite assets to PNGs or deterministic placeholders, a day/night and depth-aware parallax background renderer for forest, night forest, magical grove, cave, and deep cave scenes, and a chunk render command cache with visible/rebuilt/evicted draw metrics plus a settings-driven cache budget. Terrain tile commands now carry autotile masks into sprite-frame selection, and dirt/grass/stone/wood/leaves use 16-mask blockface source frames plus matching AI sprite briefs for 256x16 terrain strips. The character/ambient-life asset wave now includes player action frames, v2 body/hair/clothing/accessory sheets, rabbit, bird, bat, cave worm, foliage props, furniture/decor/tool/UI/icon assets, corrected 512x128 parallax backgrounds, and first magic item/projectile sprites.

World generation is modular and deterministic. It now generates surface terrain, dirt/stone layers, cave walkers, large organic caverns with connector tunnels, profile-driven ore veins, shaped surface lakes, cave pools, underground walls with cave-core cleanup, simple structures, pass-through mineable trees, a forest biome map, and a safe spawn point. `WorldGenerationProfile` exposes validated tuning for each of these passes, while `WorldAnalyzer` reports cavern regions, liquid bodies, surface/cave liquid, and wall coverage. `InfiniteWorldChunkGenerator` streams deterministic chunks in negative and positive X while keeping vertical bounds finite and dimension-banded. `ChunkStreamingService` applies a per-update operation budget, prioritizes nearby loads and distant unloads, reports deferred backlogs, saves dirty chunks safely, and publishes typed lifecycle events.

The engine also has increasingly useful debug foundations: `EngineDebugSnapshotBuilder`, `/debug world`, `WorldAnalyzer`, dirty-region tracking, global event observation, a bounded `GameEventJournal`, an engine-neutral `PerformanceProfiler` with rolling averages/peaks/budgets/allocation samples, save result events, world raycasts, line-of-sight checks, rectangle/circle/cone entity queries, and tile flood fill. The client exposes live pass timings, allocation rows, event history, streaming backlogs, and existing world/render/save metrics through optional overlays.

YjsE now has explicit non-Terraria genre modules. `Game.Core.Farming` supports Stardew-like loops with crop JSON definitions, seed-to-crop lookup, tilled/watered plot state, planting, water-gated daily growth, seasonal restrictions, harvesting, regrow crops, inventory integration, explicit failure statuses for UI feedback, farm plot save/load, and selected-item farming action routing. `TopDownMovementController` adds gravity-free movement over the same tile collision model. `Game.Core.Maps` adds fixed topdown map definitions with layers, collision, objects, interactables, spawn points, warps, registry validation, content/mod loading, map sessions, map-specific movement, runtime object state, interaction targeting, object action resolution, and transition services for RPG/life-sim maps. `Game.Core.Dialogue` and `Game.Core.Shops` provide the first NPC/social/economy content layer with graph validation, session advancement, shop stock, sell prices, and inventory-backed buy/sell transactions.

## Partial Systems

Rendering needs the next production pipeline: texture atlases, render-target batching, real terrain sprite sheets, proper shader passes, broader entity animation controllers, particle rendering, and deeper timing metrics. Placeholder tile and light rendering is now negative-X aware for horizontally infinite worlds, real tile/item/entity/background PNGs can be resolved through `ClientTextureRegistry` once generated, autotile-aware source-frame lookup is available, static tile draw commands are cached per chunk until mesh-dirty invalidation, and `SpriteAssetAuditService` can now check whether generated PNG files match the manifest before packaging.

World simulation has liquid data, baseline liquid flow, dirty-region scheduling, event-driven tile mutation tracking, integration into `GameSimulation`, and chunk metadata refreshes. Dirty-region clamping now preserves negative X in horizontally infinite worlds, and liquid seeding scans loaded chunks instead of pretending an infinite world has a finite horizontal span. It still needs pressure/settling behavior, richer liquid rules, particle scheduling, renderer-cache integration, and better visual water surfaces.

Interactions can target and use selected hotbar items, including pass-through mineable trees and non-solid placeables such as workbenches. Item use now returns explicit succeeded/in-progress/blocked states, failure reasons, cooldown telemetry, mining progress, restores, and applied effects. Mining publishes deduplicated start/progress/completion/blocked events; building exposes detailed placement results; consumables apply data-driven health, mana, and status effects only when useful. The client renders mining progress at the target, cooldown/blocked feedback, buffs/debuffs, and pickup magnet behavior. Remaining work is richer placement anchors, sideview tile entities/furniture, liquid rules, and server-authoritative validation.

Farming is now wired into the live client path for hoe, watering can, seed use, placeholder plot/crop drawing, daily growth, and autosave/load. It still needs proper sprite crop rendering, farm action UI feedback, weather/rain watering, shops, shipping bins, NPC schedules, and client/tooling support for authored topdown farm/town maps.

Combat can damage enemies with melee/projectiles and resolve loot, and selected item use can spawn projectile attacks with ammo checks. Magic weapons can cast `DamageType.Magic` projectiles, consume mana, and scale damage/cost from the player's current stat block. Item, projectile, and enemy definitions can apply status effects on hit/contact, and melee weapons can use data-driven rectangle, circle, or cone attack shapes. Runtime equipment/accessory stats now affect max health, max mana, movement speed, mining speed, melee damage, ranged damage, magic damage, mana cost, and mana regeneration. It still needs line-of-sight in every runtime path, reusable attack animation phases, ammo preference rules, better knockback, richer hit reactions, and combat events for UI/audio/particles.

World generation still needs biome transition bands, biome-specific wall/foliage materials, structure spacing rules, chest rooms and loot placement, seed retry integration, preview tooling, and sampled infinite-world quality gates. Structure placement routes through the tile batch mutation path, reports changed bounds and dirty chunks, and can place templates across negative X in horizontally infinite worlds.

UI has a functional debug/HUD baseline, main menu, world select, create world, loading flow, pause/settings menu, shared minimalist theming, UI animations, inventory/crafting/equipment surfaces, mana and status-effect HUD, mining/cooldown/action feedback, a persisted F2 character editor, performance and event-journal overlays, and a core UI toolkit for layout, focus, hit-testing, modal layers, and tooltips. It still needs core-driven client widgets, typed fields beyond world creation, richer debug windows, crafting search/pinning, and gamepad navigation.

## Important Missing Engine Pieces

- More tile entity types for crafting stations, doors, signs, machines, wiring, and persistent interactables.
- Data-driven tool actions so new tools/weapons/items can be added without code switches.
- Save migrations and save integrity reports.
- Save rotation, backup-before-migration, and save recovery diagnostics.
- RGB colored lighting and dynamic lights attached to entities/items/projectiles.
- AI sensors and behavior composition for line of sight, hearing, ledges, hazards, and group behavior.
- Physics materials for platforms, ladders, liquids, slopes later, damage blocks, and one-way collision.
- A trigger/zone system for scripted events, weather, encounters, rooms, and mod hooks.
- Client-facing topdown/life-sim rendering, concrete shop/dialogue/container screens, NPC schedules, authored content tooling, and farm/town map editors.
- Content browser/editor tooling for tiles, items, recipes, entities, effects, loot, and worldgen presets.
- Deterministic replay harness plus long-session profiler history/export and deeper subsystem instrumentation.
